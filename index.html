    Introduction

    Welcome to the official Ryver API documentation. Our APIs allow 3rd parties to integrate to Ryver with greater flexibility than what is supported by services such Zapier. Ryver’s APIs are a subset of OData 2.0 and will be familiar to someone with experience using REST HTTP APIs.

    The Basics

    The Ryver API is predominantly resource-oriented with a small number of RPC-style end points to perform more complex actions. Cross-origin resource sharing is supported to provide access to our API from web clients. JSON is returned for all resource requests.

    
http://host/path/odata.svc/users(1)?$top=2&$orderby=Name
\________________________/\_______/ \__________________/
             |                |               |
         base path      resource path    query options

    Authentication

    To authenticate using curl:

    curl -X POST -u user:pass "https://test.ryver.com/api/1/odata.svc/Session.Login()"

    Typical response

    {
  "username": "Stu",
  "id": 75,
  "sessionId": "develop:75:6b2e6516971b23910d75ef91b43fe7eaf3706675",
  "instanceId": "develop",
  "newUser": false,
  "__descriptor": "Stuart Carnie"
}

    A consumer must first authenticate with the Ryver API using HTTP Basic authentication over HTTPS, by calling the /api/1/odata.svc/Session.Login() API.

    Upon successful verification of credentials, the server will respond with information about the authenticated user, including a sessionId that should be used in subsequent requests.

    The sessionId should be paired with a header named Contatta-Session as follows

    Using the Contatta-Session header

    curl "http://develop.contatta.vm/api/1/odata.svc/Session.GetCurrentUser()" \
     -H "Contatta-Session: develop:75:6b2e6516971b23910d75ef91b43fe7eaf3706675"

    Chat

    Introduction

    The Chat APIs allow you to integrate and build applications to extend Ryver’s base functionality.

    Messages

    Messages are sent and received over the real-time messaging API. All messages must have a type property that indicates the type of message and how a consumer should interpret the remaining structure. The following table lists the standard messages types.

    A client can include an id property to request the server send an ack message once it has been received and successfully processed. See the ack message for more information.
    Type Description
    auth authenticate a new websocket connection
    chat represents a chat sent to a user or team
    presence_change indicates a change in presence
    user_typing indicates a user is typing
    ack response from the server to acknowledge an message sent by a client
    ping send a ping to the server
    error sent by the server in response to an invalid message

    auth message

    Client → Server: authenticating with server

    {
  "type": "auth",
  "authorization": "Session <session id>"
}

    The auth message is sent by the client to establish an authenticated connection with the server. The <session id> is obtained when calling Session.Login()

    Properties

    Name Req Type Description
    authorization ✓ string "Session" SP session-id
    agent string Name of application or client
    resource string Unique value to identify the connection

    chat message

    Server → Client: a typical chat message sent to a client

    {
  "type": "chat",
  "from": "acme+edc02251076b4190830f30f48eff8266@xmpp.ryver.com",
  "to":   "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com",
  "text": "hello world"
}

    The chat message is sent by a client and delivered to the the appropriate user or team. The to property identifies the destination of the message.

    Properties

    Name Req Type Description
    text ✓ string The text of the message
    to ✓ string The destination JID of this message
    extras object Optional extras, that can be used by integrations

    presence_change message

    Client → Server: a global presence change

    {
  "type":     "presence_change",
  "presence": "(available|away|unavailable)"
}

    Server → Client: notify clients of presence change

    {
  "type":     "presence_change",
  "from":     "acme+edc02251076b4190830f30f48eff8266@xmpp.ryver.com",
  "to":       "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com",
  "presence": "away"
}

    The presence_change message is sent by the client to indicate a user’s change of availability. Clients which receive this message can update the status in their local list of users. The to property is either directed to the user or to a team JID

    Properties

    Name Req Type Description
    presence ✓ string "available" / "away" / "unavailable"

    user_typing message

    Client → Server: indicating the user is typing

    {
  "type": "user_typing",
  "to":   "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com"
}

    Server → Client: indicating user is typing to a specific JID

    {
  "type": "user_typing",
  "from": "acme+edc02251076b4190830f30f48eff8266@xmpp.ryver.com",
  "to":   "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com"
}

    The user_typing message is sent by the client when a user starts typing. A client can send this message multiple times. The to JID indicates to which user or team the users is typing

    Clients receiving this message can show an indicator that the user is typing. The client would remove this indicator after a short delay or when a message is received.

    ack message

    Client → Server

    {
  "type": "message",
  "to":   "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com",
  "text": "hello world",
  "id":   "23TplPdS"
}

    Server → Client: in response to the previous message

    {
  "type":       "ack",
  "reply_type": "message",
  "reply_to":   "23TplPdS"
}

    The ack message is sent by the server to acknowledge a previous message sent by a client. A client requests interest in receiving an ack by including an id property.

    Not all messages require an ack. An message such as user_typing can fail and there is no resulting data loss.

    id should be a unique string or incrementing number; any other value is considered an error.

    ping message

    Client → Server

    {
  "type": "ping",
  "id":   "23TplPdS"
}

    Server → Client

    {
  "type":       "ack",
  "reply_type": "ping",
  "reply_to":   "23TplPdS"
}

    The ping message is sent by the client to check the health of the connection and server.

    The id property should be included to receive an ack from the server.

    error message

    Example error message in response to a failed auth message

    {
  "type": "error",
  "code": "auth_failed",
  "text": "authentication failed; invalid session"
}

    Error messages are sent in response to a failed request or invalid message payload.

    Properties

    Name Req Type Description
    code ✓ string a code identify the error
    text ✓ string a descriptive message of the error in English

    Error codes

    Code Description
    invalid_arg the specified argument was missing or invalid
    auth_failed the attempted auth message failed
    server_error a server error occurred
    invalid_message_type the specified message type was invalid
