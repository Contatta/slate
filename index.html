    Introduction

    Welcome to the official Ryver API documentation. Our APIs allow 3rd parties to integrate to Ryver with greater flexibility than what is supported by services such Zapier. Ryver’s APIs are a subset of OData 2.0 and will be familiar to someone with experience using REST HTTP APIs.

    The Basics

    The Ryver API is predominantly resource-oriented with a small number of RPC-style end points to perform more complex actions. Cross-origin resource sharing is supported to provide access to our API from web clients. JSON is returned for all resource requests.

    
http://host/path/odata.svc/users(1)?$top=2&$orderby=Name
\________________________/\_______/ \__________________/
             |                |               |
         base path      resource path    query options

    Authentication

    To authenticate using curl:

    curl -X POST -u user:pass "https://test.ryver.com/api/1/odata.svc/Session.Login()"

    Typical response

    {
  "username": "Stu",
  "id": 75,
  "sessionId": "develop:75:6b2e6516971b23910d75ef91b43fe7eaf3706675",
  "instanceId": "develop",
  "newUser": false,
  "__descriptor": "Stuart Carnie"
}

    A consumer must first authenticate with the Ryver API using HTTP Basic authentication over HTTPS, by calling the /api/1/odata.svc/Session.Login() API.

    Upon successful verification of credentials, the server will respond with information about the authenticated user, including a sessionId that should be used in subsequent requests.

    The sessionId should be paired with a header named Contatta-Session as follows

    Using the Contatta-Session header

    curl "http://develop.contatta.vm/api/1/odata.svc/Session.GetCurrentUser()" \
     -H "Contatta-Session: develop:75:6b2e6516971b23910d75ef91b43fe7eaf3706675"

    Chat

    Introduction

    The Chat APIs allow you to integrate and build applications to extend Ryver’s base functionality.

    Events

    Events are sent and received over the real-time messaging API. The following events are sent and received over the real-time API

    Type Description
    auth authenticate a new websocket
    message represents a message sent to a user or team
    presence_change indicates a change in presence
    user_typing indicates a user is typing
    ack response from the server to acknowledge an event sent by a client

    auth event

    Client → Server: authenticating with server

    {
  "type": "auth",
  "authorization": "Session <session id>"
}

    The auth event is sent by the client to establish an authenticated connection with the server. The <session id> is obtained when calling Session.Login()

    message event

    Server → Client: a typical message sent to a client

    {
  "type": "message",
  "from": "acme+edc02251076b4190830f30f48eff8266@xmpp.ryver.com",
  "to":   "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com",
  "text": "hello world"
}

    The message event is sent by a client and delivered to the the appropriate user or team. The to property identifies the destination of the message.

    presence_change event

    Client → Server: a global presence change

    {
  "type":     "presence_change",
  "presence": "(available|away|unavailable)"
}

    Server → Client: notify clients of presence change

    {
  "type":     "presence_change",
  "from":     "acme+edc02251076b4190830f30f48eff8266@xmpp.ryver.com",
  "to":       "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com",
  "presence": "away"
}

    The presence_change event is sent by the client to indicate a user’s change of availability. Clients which receive this event can update the status of their local list of users. The to property is either directed to the user or to a team JID

    user_typing event

    Client → Server: indicating the user is typing

    {
  "type": "user_typing",
  "to":   "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com"
}

    Server → Client: indicating user is typing to a specific JID

    {
  "type": "user_typing",
  "from": "acme+edc02251076b4190830f30f48eff8266@xmpp.ryver.com",
  "to":   "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com"
}

    The user_typing event is sent by the client when a user starts typing. A client can send this event multiple times. The to JID indicates to which user or team the users is typing

    Clients receiving this message can show an indicator that the user is typing. The client would remove this indicator after a short delay or when a message is received.

    ack event

    Client → Server

    {
  "type": "message",
  "to":   "acme+27053427bb9f4c448d79ef5fcb4da78c@xmpp.ryver.com",
  "text": "hello world",
  "id": 1
}

    Server → Client: in response to the previous event

    {
  "type":     "ack",
  "reply_to": 1
}

    The ack event is sent by the server to acknowledge a previous event sent by a client. A client requests interest in receiving an ack by including an id property in an event. Not all events require an ack, such as user_typing, given a failed delivery will not cause data loss. If a message event fails, the client would attempt to retry sending the message or display an alert to the user.

    id should be a string or number; any other value is considered an error. It is recommended that the id is either an incrementing number or a UUID.

    Errors

    This error section is stored in a separate file in `includes/_errors.md`. Slate allows you to optionally separate out your docs into many files…just save them to the `includes` folder and add them to the top of your `index.md`’s frontmatter. Files are included in the order listed.

    The Kittn API uses the following error codes:

    Error Code Meaning
    400 Bad Request – Your request sucks
    401 Unauthorized – Your API key is wrong
    403 Forbidden – The kitten requested is hidden for administrators only
    404 Not Found – The specified kitten could not be found
    405 Method Not Allowed – You tried to access a kitten with an invalid method
    406 Not Acceptable – You requested a format that isn’t json
    410 Gone – The kitten requested has been removed from our servers
    418 I’m a teapot
    429 Too Many Requests – You’re requesting too many kittens! Slow down!
    500 Internal Server Error – We had a problem with our server. Try again later.
    503 Service Unavailable – We’re temporarially offline for maintanance. Please try again later.
